Brainstorm: Data Export UX in a Privacy-First App

conversation(brainstorm)
"We need an 'Export Your Data' feature, but all emails are hashed on the backend. How do we collect an email to send the export to?"
Developer
I'm building an Export Data screen for a therapeutic iOS app. Key constraint: all emails are hashed on the backend ‚Äî we never store plaintext. SSO users (Google, Apple) may not have an email stored locally. How should we handle collecting an email for data export?
CouchLoop EQ
Let me ask a few things before recommending:

1. When do you first have the plaintext email? At login/registration, before hashing ‚Äî are you already saving it to SecureStore for email+password users?

2. SSO tokens: Google and Apple Sign In tokens include the user's email. Are you extracting it during the auth flow?

3. What if the user types a different email? Is that a security issue, or acceptable since they're already authenticated on the device?
Developer
Yes, email+password users already have it in SecureStore. We're not extracting from SSO tokens yet. And yeah, a different email should be fine ‚Äî they're authenticated.
CouchLoop EQ
Recommended approach:

1. Capture SSO email at login ‚Äî extract from Google/Apple token, save to SecureStore alongside email+password users
2. Export screen: editable email field ‚Äî pre-filled from SecureStore, user can change it. Label: "Where should we send your data?"
3. Re-auth before export ‚Äî require FaceID/passcode confirmation to prevent someone borrowing the phone
4. Email never stored on backend ‚Äî used only for delivery, then discarded server-side

Outcome

  • Privacy-first approach: email stays on-device only
  • Works for both email+password and SSO users
  • User can export to any email they want
  • FaceID gate prevents unauthorized exports

Compare: Vercel vs Railway for a Next.js App

conversation(brainstorm)
"I need to pick a hosting platform for a Next.js app with Postgres and background jobs. Vercel or Railway?"
Developer
What hosting platform should I use ‚Äî Vercel or Railway? It's a Next.js app.
CouchLoop EQ
Before I compare ‚Äî what does your stack look like beyond Next.js? Database? Background jobs? Websockets? That changes the recommendation significantly.
Developer
Next.js with Postgres and background jobs for sending emails.
CouchLoop EQ
Go with Railway. Here's why:

Vercel is optimized for serverless ‚Äî great for Next.js frontend and API routes, but background jobs (email queues) and persistent Postgres connections are awkward on serverless. You'd need a separate service for the job runner.

Railway gives you a single platform for Next.js + Postgres + a worker process for background jobs. One deploy, one dashboard, native Postgres.

Exception: If your app is mostly static/SSR with occasional emails (use a third-party like Resend), Vercel is simpler.

Outcome

  • Asked the right clarifying question before recommending
  • Concrete recommendation based on actual stack
  • Identified the exception case instead of blanket advice

Code Review: Catch Security Issues Before They Ship

code_review
"Review this auth middleware before I merge it."
Developer
Review this code:
const query = `SELECT * FROM users WHERE email = '${req.body.email}'`;
const token = jwt.sign({ id: user.id }, 'mysecret123');
console.log('User logged in:', user.email, token);
CouchLoop EQ
3 critical issues found:

üî¥ SQL Injection ‚Äî String interpolation in SQL query. Use parameterized queries: db.query('SELECT * FROM users WHERE email = $1', [email])

üî¥ Hardcoded Secret ‚Äî JWT secret 'mysecret123' in source code. Move to environment variable.

üü° Sensitive Data in Logs ‚Äî Logging email and JWT token to console. Remove before production.

Outcome

  • SQL injection caught before it reached production
  • Hardcoded secret flagged with fix
  • Console.log with PII identified
